One thing we haven't discussed from an architecture perspective is an admin console. We don't need anything crazy fancy at the start, but several things stand out as being necessary for a v1 product: 1) pre-appproval of email addresses as a prerequisite for account creation (or required post-creation approval by an admin) because we don't necessarily want anyone on the intranet to be able to create and account and provision resources; 2) change user quotas, or possibly switch between two levels of non-admin accounts: `standard` and a power user, which would allow us to easily provide/modify group-level permissions and/or quotas; 3) manage or initiate resetting of user passwords; 4) view and manage all jobs across all users (include jobs that have been "deleted" by users, which shouldn't actually delete the job but should instead just remove the job from the user's job history); etc. Any other obvious admin-level tasks that the v1 console should have?
