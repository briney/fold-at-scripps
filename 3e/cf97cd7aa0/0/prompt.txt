**1. [Important] Redaction regex under-masks passwords containing `@` â€” partial password leaks into redacted output.**
`src/fold_at_scripps/foldapp/envfile.py:33` â€” regex is `r"://([^:/@]+):[^@]*@"`. The password-matching group `[^@]*` is non-greedy against `@` by construction, so if the password itself contains an `@` (a character that is legal, if unusual, in a URL-encoded or raw postgres password), the substitution only masks up to the first `@` and leaves the remainder of the password exposed in the "redacted" output. Verified:
```

```
This is exactly the failure mode the reviewer brief calls out ("note if it mishandles a URL with... special chars"). Since `redact_settings` exists specifically to keep secrets out of logs/API responses, a partial leak defeats its purpose for any password containing `@`. Likely low real-world frequency (most generated/deployed DB passwords won't contain `@`), but it is a correctness gap in a security-sensitive function with no test coverage for this case. This exact regex is also what the brief's own Step 3 reference snippet specifies, so this is **plan-mandated** â€” the implementer copied the brief's regex verbatim rather than hardening it.
Fix: match greedily to the last `@` before the host, e.g. `r"://([^:/@]+):[^@]*@(?=[^@]*$)"` or better, use `urllib.parse.urlsplit`/`urlunsplit` to parse the netloc properly instead of a hand-rolled regex, which also avoids issues if the URL has no scheme separator quirks.
