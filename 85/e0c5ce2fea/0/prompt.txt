The `default=False` is a Python/ORM-side default. It works for ORM-constructed objects, but if a row is ever inserted via raw SQL (migrations, fixtures, manual psql) it will `NULL` if the column has no `server_default`. The safe fix is to add `server_default=false()` (importing `false` from `sqlalchemy`) alongside the ORM `default`:

```python
from sqlalchemy import Boolean, CheckConstraint, DateTime, Integer, false, func

maintenance_mode: Mapped[bool] = mapped_column(
    Boolean, default=False, server_default=false(), nullable=False
)
```

This is a should-fix, not critical, because all current insert paths go through the ORM.

### Strengths

- **Session isolation is correct**: `_dispatch` uses `async with self._sessionmaker() as session` — each concurrent task gets its own transaction. This is the right pattern for concurrent async SQLAlchemy work and prevents session-sharing bugs that would cause hard-to-diagnose failures under load.
- **Mutation-safe reap**: `list(self._inflight.items())` defensively copies the dict view before iterating and deleting — avoids `RuntimeError: dictionary changed size during iteration`.
- **`return_exceptions=True` in drain**: task failures do not prevent GPU release; `_reap` runs after gather regardless, so GPUs are always freed on drain even when tasks fail.
- **`run_once` session scope is correct**: the claim loop runs inside a single session (good for transactional claim semantics), while execution happens in separate per-dispatch sessions (good for concurrency). These are the right boundaries.
- **No GPU leak on failed dispatch**: if `execute_run` raises, the asyncio task transitions to done with an exception; `_reap` checks `task.done()` (which is True for cancelled/failed tasks, not just successful ones), so GPUs are released correctly regardless of outcome. This is the critical correctness property for a GPU scheduler.

### Issues

#### Critical (Must Fix)

#### Important (Should Fix)

**`service.py:86-91` — silent no-op if run is not found in `_dispatch`**

```python
run = await session.get(Run, run_id)
if run is not None:
    await execute_run(session, run, self._executor, self._storage)
```

If `session.get` returns `None` (row deleted between claim and dispatch — shouldn't happen in normal operation, but can happen in tests or under unusual DB conditions), the run stays in `RUNNING` status forever (it was claimed by `claim_runnable_run` before `_dispatch` was called), and its GPUs are still released by `_reap` once the task finishes. The run becomes a zombie: `RUNNING` with no task. This won't cause a GPU leak, but it will cause a stuck run that can never be re-queued. A `logging.warning` or an explicit status reset to `FAILED` would be appropriate here.

**`service.py:79-84` — `_reap` does not log task exceptions**

`drain` uses `return_exceptions=True`, which is correct, but neither `drain` nor `_reap` logs or surfaces exceptions from failed tasks. If `execute_run` raises unexpectedly, the failure is silently swallowed. In production this would make debugging very difficult. At minimum, `_reap` should inspect `task.exception()` for done-but-not-cancelled tasks and log it.

#### Minor (Nice to Have)

**`service.py:108-113` — `drain` snapshot vs. new arrivals**

`drain` snapshots `self._inflight.values()` at call time. If `run_once` is called concurrently while `drain` is awaiting (unlikely in current usage, but possible with `run_forever`), tasks added after the snapshot are not awaited by the current `drain` call and will be reaped on the next `_reap` call. The brief's intent is graceful shutdown, and in the tests drain is always called after `run_once` completes, so this is not a practical issue — just worth noting if `drain` is ever exposed as a true shutdown hook in a concurrent context.

**`tests/scheduler/test_service.py:169` — `test_run_once_dispatches_up_to_capacity` relies on timing-sensitive ordering**

The test asserts `pool.available == 2` after `drain()`, which is correct. But the second `run_once()` call relies on `drain()` having already updated `_inflight` (via `_reap`) so that there are no stale inflight entries blocking the third claim. This works correctly because `drain` calls `_reap` as its last step. This is fine, just not immediately obvious to a reader — a brief comment explaining why `drain` reaps would help.
